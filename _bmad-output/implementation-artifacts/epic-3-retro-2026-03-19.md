## What Went Well

1. **Clean incremental story design** — Story 3-1 laid the adapter foundation, 3-2 built trigger intelligence, 3-3 added dedup sophistication. Each story was immediately useful on its own with zero rework.

2. **Code reviews caught 2 critical bugs:**
   - H1 in Story 3-2: `notification.backlog` self-consumption causing infinite recursive loop — would have been a production disaster
   - M3 in Story 3-2: `String()` casts producing `[object Object]` in notification messages — replaced with `typeof` guards

3. **100% Epic 2 retro action item follow-through** — All 4 action items resolved:
   - wireDetection decomposition (extracted `wireNotificationServices()`)
   - Event type taxonomy (DEFAULT_TRIGGER_MAP with 10 explicit types)
   - Interface mock validation (mocks already use typed returns)
   - Notifier plugin audit (all 4 plugins audited, 3 adapters created)

4. **Forward-compatible infrastructure** — Trigger map and digest mode handle future event types automatically. When Epic 4 adds `agent.offline` and `eventbus.backlog` publishers, notifications work without touching notification-service.ts.

5. **Explicit deferred item tracking** — Every story's "Limitations (Deferred Items)" section with Status/Requires/Epic/Current fields made cross-story planning straightforward.

6. **Inter-story knowledge transfer** — ESLint split-edit workaround discovered in 3-1, applied in 3-2, became standard practice by 3-3. Dev Notes sections carry learnings forward.

## What Could Improve

1. **Adapter utility duplication** — `mapPriority()` and `toOrchestratorEvent()` were copy-pasted identically across 3 plugin adapters (slack, webhook, composio). Should have been extracted into shared utility in Story 3-1, not retroactively. **RESOLVED during retro:** extracted to `@composio/ao-core/notification-adapter.ts`.

2. **wireDetection complexity** — Notification wiring added to the already-crowded wireDetection function. Made the function harder to reason about. **RESOLVED during retro:** extracted `wireNotificationServices()`.

3. **Forward-compatible code lacks integration tests** — `agent.offline`, `eventbus.backlog` triggers and digest mode (`medium` priority) exist in code but have no publishers yet. Can't integration-test until Epic 4 adds them.

4. **ESLint hook friction** — Split edits (import in one edit, usage in another) blocked by lint hook in all 3 stories. Adapted by combining in single edits. Minor friction, not a process failure.

## Technical Concerns for Epic 4

1. **Circuit breaker for notification delivery.** If Slack/webhook endpoints are down, NotificationService currently retries 3 times then DLQs. Epic 4's circuit breaker pattern should wrap notification plugin delivery to fast-fail after repeated failures.

2. **Health monitoring publisher gap.** Story 4-4 needs to publish `agent.offline` (distinct from `agent.blocked`) and `eventbus.backlog` events. The notification triggers are ready — publishers are the missing piece.

3. **Error classification integration.** Story 4-1's structured error logging should feed into notifications. Classified errors at `critical` severity should trigger immediate push notifications through the existing trigger map.
